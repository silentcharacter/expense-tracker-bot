# Постановка задачи: фича «Portfolio Report»

## Контекст

В проекте уже есть Telegram-бот с Telegram Mini App и бэкенд на Cloud Function, который обслуживает мини-апп (валидация `initData`, доступ к Secret Manager уже реализованы). Нужно добавить **новую фичу**, не заводя отдельный сервис.

## Цель

По запросу пользователя из меню мини-аппа бэкенд в рантайме собирает актуальный состав крипто-портфеля из нескольких внешних источников, нормализует его и возвращает **структурированный HTML-отчёт**, который мини-апп показывает как отдельный экран.

Проблема, которую решаем: стандартные трекеры не видят активы, заблокированные в торговых ботах и в Earn-продуктах. Эта фича должна видеть их явно.

## Архитектурный подход

- Реализовать как **новый роут внутри существующего бэкенда** (тот же Cloud Function), а не как отдельный сервис. Причина: запрос остаётся same-origin, переиспользуются уже готовые аутентификация (`initData`) и доступ к секретам, не нужен второй CORS-origin и второй деплой.
- Триггер — **on-demand**: отдельный HTTP-вызов из меню мини-аппа. Это не cron и не фон. Вызов идёт мимо обработчика Telegram-вебхуков, поэтому его латентность не блокирует обработку сообщений.
- Эндпоинт отдаёт отчёт в двух представлениях из одного сборщика данных: `html` (страница для экрана мини-аппа) и `json` (на случай краткой сводки в чат). Конкретную структуру и вёрстку отчёта оставляем на отдельное обсуждение — **в этой постановке они не фиксируются**.

## Главное: подход к обращению к внешним источникам

Источников три, и обращаемся к ним тремя разными способами:

- **Binance** — через Binance Spot REST API (подписанные приватные эндпоинты).
- **Bybit** — через Bybit API V5 (подписанные приватные эндпоинты).
- **Trust Wallet** — НЕ через API кошелька (его нет), а через чтение публичных адресов кошелька напрямую из блокчейнов.

Общие принципы для бирж:

- **Только чтение.** API-ключи бирж — строго read-only, без прав на трейд и вывод.
- **Тонкие подписанные REST-вызовы, а не тяжёлые SDK.** Обращения к биржам — это подписанные GET-запросы; полноценные клиентские библиотеки тащить не нужно, чтобы не раздувать деплой и cold start общего бэкенда. Достаточно HTTP-клиента и хелпера для подписи.
- **Параллельность.** Все обращения к источникам выполнять конкурентно (несколько REST-вызовов на источник), чтобы суммарная латентность отчёта оставалась в пределах нескольких секунд.
- **Секреты — в Secret Manager**, переиспользуя существующий механизм бэкенда.
- **Устойчивость к частичной недоступности.** Если один источник недоступен или вернул ошибку, отчёт должен формироваться по остальным и явно помечать недостающий блок, а не падать целиком.
- **Нормализация и оценка.** Активы из всех источников приводятся к единому виду и оцениваются в USD по текущим рыночным ценам (публичные тикеры бирж).

### Binance

**API:** Binance Spot REST API (база `api.binance.com`). **Аутентификация:** подписанные запросы — HMAC-SHA256-подпись по строке параметров, API-ключ в заголовке `X-MBX-APIKEY`, обязательный `timestamp` (+ `recvWindow`).

Активы разнесены по двум семействам эндпоинтов, поэтому собрать нужно **три слоя**:

1. **Spot-баланс** (семейство `/api/v3/*`) — свободные и заблокированные средства аккаунта.
2. **Simple Earn** (семейство `/sapi/v1/simple-earn/*`) — отдельные позиции Flexible и Locked. Это та часть, которую обычные трекеры теряют; её надо запрашивать явными эндпоинтами Earn, в spot-балансе её нет.
3. **Открытые ордера** (`/api/v3/*`) — для атрибуции заблокированных средств.

Важный нюанс подхода: **у нативных торговых ботов Binance (Spot Grid / DCA) нет публичного API**. Их средства физически лежат в spot-аккаунте как `locked` и видны через открытые ордера, но разбивки «сколько в каком боте» через официальный API получить нельзя. Агент должен учитывать эти средства на уровне spot-баланса, а не пытаться найти несуществующий эндпоинт ботов.

### Bybit (API V5, единый Unified-аккаунт)

**API:** Bybit API V5 (база `api.bybit.com`). **Аутентификация:** подписанные запросы — HMAC-SHA256-подпись по `timestamp + apiKey + recvWindow + queryString`, заголовки `X-BAPI-*`. Собрать нужно **несколько модулей**:

1. **Wallet balance** — баланс единого аккаунта по монетам.
2. **Earn / staking позиции** — отдельный модуль Earn (Flexible Saving и т.п.), как и на Binance, в общий баланс не входит.
3. **Открытые позиции** — для деривативной части.
4. **Торговые боты (Spot Grid / DCA)** — в отличие от Binance, у Bybit **есть отдельный Trading Bot API**. Средства бота сегрегированы внутри него и не всегда видны в обычном wallet-balance, поэтому ботов нужно **опрашивать отдельным запросом** и складывать их состав в общий портфель. Учесть, что Bot API использует иной формат ответа (`status_code`/`debug_msg`), чем основной V5.

Эксплуатационный нюанс, который агент должен заложить: **read-only ключ Bybit без привязанного IP истекает через ~90 дней**. Это не блокер для разработки, но в документации фичи и в обработке ошибок аутентификации это надо отразить (понятное сообщение о протухшем ключе, а не «непонятная ошибка»).

### Холодный кошелёк (Trust Wallet)

**Важно:** Trust Wallet — это некастодиальный мультичейн-кошелёк (100+ сетей: EVM, Bitcoin, Solana и др.). У него **нет аутентифицируемого API портфеля** — некуда «логиниться», приватные ключи нигде не используются и не передаются. Сам Trust Wallet (Wallet Core) чтение балансов не предоставляет и отсылает к RPC/API конкретного блокчейна.

Поэтому подход принципиально иной, чем у бирж: **watch-only по публичным адресам кошелька, отдельно для каждой сети.** Один seed Trust Wallet порождает разные адреса в разных семействах блокчейнов, поэтому это не один адрес, а набор «сеть → публичный адрес».

Для каждой сети из набора:

- **EVM-сети** (Ethereum, BNB Smart Chain, Polygon и т.п.) — нативный баланс и балансы токенов через RPC (`eth_getBalance`, вызовы контрактов токенов) или через explorer-API соответствующей сети (Etherscan / BscScan / Polygonscan).
- **Bitcoin** — по публичному адресу или `xpub` через блок-эксплорер (например, mempool.space).
- **Solana** и прочие не-EVM сети — через их публичный RPC / индексер.

**Рекомендация по подходу:** поскольку источник мультичейновый, разумнее не подключать каждую сеть руками, а взять **один мультичейн-индексер балансов** (например, Trust Wallet API Gateway, либо Covalent/GoldRush, Moralis, Ankr, Alchemy) — он по одному адресу отдаёт нативные и токеновые балансы по многим сетям в едином формате. Это сильно упрощает покрытие ERC-20 / BEP-20 / SPL-токенов. Выбор конкретного провайдера — на этап реализации.

Набор «сеть → публичный адрес» вынести в конфиг/секреты, не хардкодить. Полученные балансы нормализуются и оцениваются в USD так же, как биржевые.

## Что в эту постановку НЕ входит (обсуждается с агентом отдельно)

- Конкретная структура и разделы отчёта.
- HTML-вёрстка и оформление (подхват темы мини-аппа и т.п.).
- Формат краткой текстовой сводки для чата.
- Точная схема нормализованной модели данных.

## Нефункциональные требования и ограничения

- **Безопасность:** эндпоинт защищён существующей аутентификацией мини-аппа (`initData`); ключи read-only; URL отчёта и секреты не логировать.
- **Латентность:** допустимо несколько секунд на холодном старте; обязательна параллельность вызовов.
- **Отказоустойчивость:** частичный сбой источника не валит весь отчёт.
- **Минимизация зависимостей** ради cold start общего бэкенда.

## Открытые вопросы для уточнения с заказчиком

1. На какой площадке работает DCA/Martingale-бот «Infantry» — нативный Bybit, нативный Binance или сторонняя платформа (3Commas/Pionex/Bitsgap). От этого зависит, опрашивать ли его через Trading Bot API Bybit или учитывать как spot-баланс Binance.
2. Набор «сеть → публичный адрес» Trust Wallet: какие именно блокчейны нужно учитывать и публичные адреса (или `xpub` для UTXO-сетей) в каждом из них.
3. Желаемые разделы и уровень детализации отчёта.
